✅ We use modules in playbook files, and these are small programs (python programs) which does the actual work.

✅ We can check the module docuementation:
    # ansible-navigator doc -m stdout user
    OR use FQCN
    # ansible-navigator doc ansible.builtin.user -m stdout
    (will list all the supported attributed which are mandatory and not. Also, shows some example how to use module)

✅ few paramater accpet the value as boolean:
    True == true == Yes == yes == y == 1
    False == false == No == no == n == 0
    (any value can be used and accepted in playbook with modules, but best practice to keep the consistent, use only one across playbooks)

✅ To see the all collections available inside the Execution Environment:
    # ansible-navigator collections list

✅ A Sample Yaml File:

---                                                     <---- This indicates that starting of yaml file but comletely optional
- name: Connectivity test with debug                    <---- Name of the task, optional but good to have to describe what playbook does
  hosts: all                                            <---- hosts against this playbook will run
  gather_facts: false                                   <---- Additional paramater to change the behaviour of Ansible run

  tasks:                                                <---- From here, tasks or list of action are starting
    - name: Ping hosts                                  <---- Name of tasks, optional but good to have to describe what task does
      ansible.builtin.ping:                             <---- Module which will perform the real action
                                                        <---- Below are also tasks here with another modules, a modules can have a multiple paramater in key:value format
    - name: Show hostname
      ansible.builtin.command: hostname
      register: host_output                                   

    - name: Print hostname
      ansible.builtin.debug:
        var: host_output.stdout
